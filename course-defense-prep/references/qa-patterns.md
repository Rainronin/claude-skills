# 答辩 Q&A 追问链设计模式

## 四段式结构

```
假设提问 → 标准回答 → 可能追问 → 追问应对
```

每道题必须有这四段。缺一段就漏了防御深度。

## 追问深度递进

追问1 沿技术细节深挖 → 追问2 边界/替代方案 → 追问3 底层原理

示例：
```
Q: 打乱算法怎么做？
A: 从已解状态出发随机移动 N 步，N = gridSize² × 20
追问: 为什么是 ×20 不是 ×10？
应对: 实验确定——×5 打乱不充分，×50 初始化慢但不增加效果，20 是平衡值
追问: 纯随机排列为什么不行？
应对: 只有 50% 概率有解（逆序对数为偶数），随机移动法 100% 有解且不需要验证
```

## 六大类问题的追问方向

### 基础概念类
- 追问方向：概念对比（A 和 B 什么区别）、代码体现（项目哪里用到了）、语言特性（Java 8 有什么变化）
- 目标：证明不是死记硬背，而是理解了选择场景

### 代码实现类
- 追问方向：边界处理（边缘 case 怎么处理）、性能（如果数据量大 10 倍）、替代方案（为什么不用 XXX 方法）
- 目标：证明考虑了工程实践，不是玩具代码

### 设计思路类
- 追问方向：架构对比（和 MVC/MVP 什么区别）、跨平台复用（Web 端能用吗）、团队协作（如果 3 人开发怎么分工）
- 目标：证明有架构思维，不只是堆代码

### 数据层类
- 追问方向：SQL 细节（为什么用 PreparedStatement）、持久化选择（为什么 H2 不是 MySQL）、安全（SQL 注入怎么防）
- 目标：证明理解数据库不只是 CRUD

### GUI/前端类
- 追问方向：组件对比（为什么自绘不用组件）、事件模型（EDT 是什么）、渲染性能（100 个碎片怎么优化）
- 目标：证明理解 UI 框架原理

### 开放思考/压力测试类
- 追问方向：改进方案、扩展瓶颈、重做选择、技术含量辩护
- 目标：证明能跳出项目看全局，临场应变不慌

## 压力测试题的通用应对策略

| 评委说什么 | 你的回应思路 |
|-----------|------------|
| "这个太简单了" | 表面简单但实现了完整分层架构 + 代码量 + 覆盖 N 个知识点 + 如果有时间会加什么 |
| "市面上有现成的为什么自己做" | 课程要求从零实现以覆盖知识点 + 现成工具不满足特定需求（如风格要求） |
| "你写的代码有多少是自己的" | 全部自己写 + 提交历史可查 + 第三方只有 Maven 依赖 |
| "如果重做会怎么改" | 3 个方向：架构改进 + 测试 + 新功能，体现思考深度 |
