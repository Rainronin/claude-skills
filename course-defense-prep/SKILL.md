# 课程答辩准备文档生成器

为大学课程设计项目生成一份结构化的答辩准备 DOCX 文档。核心价值：**把项目代码和评分标准转化为答辩时可直接使用的问答手册**。

## 触发方式

| 方式 | 操作 |
|------|------|
| **自动触发** | 对话中提到"答辩文档""答辩攻略""准备答辩""假设提问""defense""defense preparation"等关键词自动加载 |
| **手动命令** | 输入 `/答辩` 或 `/defense` 或 `/course-defense-prep` |

## 通用性

本 skill 适配**任何语言/框架的项目**（Java、Python、Web、HarmonyOS、小程序等），不绑定特定技术栈。下文所有技术术语均为示例，实际执行时替换为当前项目的具体技术。

## 工作流程（4 阶段）

### 阶段 1：收集信息 + PPT 前置检查

**先判断是否有 PPT 文件：**
```
1. 检查项目目录或用户指定的路径是否有 .pptx/.ppt 文件
   ├── 有 → 提取 PPT 每页的标题和要点，作为逐页提问的锚点
   └── 无 → 以项目 README + 源码 + 课程要求为主轴
```

**必须阅读的文件：**
1. 课程评分标准 / 考查要求文档（如有）— 提取评分维度、知识点要求、答辩流程
2. PPT 文件（如有）— 提取每页标题和核心要点
3. 项目 README.md / CLAUDE.md（如有）— 了解项目背景和技术栈
4. 项目关键源码 — 了解核心架构、技术决策、数据模型

**提取关键信息：**
- 评分维度及各维度分值
- 项目中的关键技术决策
- PPT 每页的功能点（如有 PPT）
- 项目包结构和分层架构

### 阶段 2：设计文档结构

按以下七章组织文档（通用模板）：

```
一、项目概述（30秒版）       ← 一句话讲清楚做了什么
二、白话知识点讲解           ← 用比喻+日常语言，面向纯萌新（≥7篇）
三、基础概念问答             ← 老师指着术语问"这是什么意思"（≥15题）
四、逐页假设提问与回答        ← 按 PPT 页或功能点组织，覆盖每一处（≥30题）
五、进阶深入提问             ← 架构决策/替代方案/扩展思路（8~12题，可选）
六、答辩评分对应方案          ← 得分策略 + 丢分警示 + 速查卡
七、演示前自检清单           ← 逐项打勾
```

### 阶段 3：编写问答内容（核心）

这是本文档区别于普通答辩 PPT 的关键。问答按三层分级组织：

#### 三层问答体系

**一层：基础概念问答（≥15 题）**
- 问法模拟："老师指着 PPT/代码里的一个术语问'这是什么意思'"
- 覆盖范围：项目中出现的所有基础术语。例如：路由是什么、回调函数是什么、Promise 是什么、什么是组件、JSON 是什么、import/export 是什么、@注解是什么、什么是持久化、Toast 是什么……
- **强制要求**：每题必须用**日常比喻**解释（禁止用术语解释术语）。每题必须关联项目中的**具体使用位置**
- 定位：让老师知道你"真懂了"，不是照搬教程

**二层：逐页提问（≥30 题）**
- 如果有 PPT：按 PPT 页码组织，格式为「PPT第X页 — 页名」→ 下面列 3~6 个可能问题
- 如果没有 PPT：按项目功能点组织（例如"首页功能""详情页""数据管理"等）
- 每题的格式：Q（老师可能的问法）+ A（标准回答，3-5句，口语化，关联代码位置）
- 定位：展示"我知道我做了什么"

**三层：进阶深入提问（8~12 题，可选）**
- 覆盖：架构决策对比、为什么选A不选B、如果重做怎么改进、扩展到N倍规模怎么改
- 定位：应对"这个有什么技术含量"类压力测试

#### 问题编写要点
- 回答必须能关联到项目中的具体代码位置（"在 XX 文件第 Y 行"）
- 每个回答控制 3-5 句话，便于临场记忆
- 基础概念题必须用比喻，不能用术语解释术语
- 压力测试题（"太简单了"、"这个有什么技术含量"）必须准备

**附加快查卡：** 在评分对策章末尾放一张三列表格："类别 | 假设提问（短版）| 一句话应答"，方便答辩前 5 分钟快速过一遍。

### 阶段 4：生成 DOCX

**前置检查：协作 Skill 探测**

在开始生成前，先检查以下两个 skill 是否可用：

| 协作 Skill | 用途 | 若可用 |
|-----------|------|--------|
| `docx` | 专业 DOCX 生成、表格格式化、样式管理 | 委托它生成最终 DOCX |
| `frontend-design` | 视觉设计、配色方案、排版美化 | 委托它设计封面配色、表格样式、标题层级视觉方案 |

**调用方式：**
- 若 `docx` 可用 → 将完整文档内容和章节结构传给 `docx` skill，由它生成最终 DOCX
- 若 `frontend-design` 可用 → 在阶段 2 设计完成后，用它生成一份"视觉设计指南"（配色、间距、字体层级、表格边框风格），再传给 `docx` skill 执行
- 两者都可用 → **三 skill 联合作战**：本 skill 负责内容策划 + Q&A 设计 → `frontend-design` 负责视觉方案 → `docx` 负责文档生成
- 都不可用 → 回退到 python-docx 脚本模式

> **重要**：不要假设 skill 一定存在。每次执行前必须检查 `available_skills` 列表。

## 文档质量标准

**白话知识点编写三原则：**
1. **用比喻** — 每个知识点配一个日常比喻（例如："@State 像报警器，变量一变 UI 就更新"）
2. **禁术语链** — 不能用术语 A 解释术语 B（例如：不能说"Promise 是一个异步操作的占位符"，应该说"Promise 像餐厅的取餐号"）
3. **关联代码** — 每个知识点必须说清楚"在我们项目的哪里用到了"

**回答质量三原则：**
1. **关联代码** — 每个技术回答必须能说"在 XX 文件第 Y 行"
2. **控制长度** — 标准回答 3-5 句
3. **口语化** — 不能像背课文，要用自己的话说

**丢分行为清单（答辩评分对策章必须包含）：**
- ❌ 回答太泛 — "用了一个框架" → 必须说"用了什么框架的什么具体功能"
- ❌ 说不清代码位置 — 必须能指出具体文件和行号
- ❌ 答不上替代方案对比 — "为什么不用 X"时必须能比较优劣
- ❌ 背课文式回答 — 必须口语化 + 带比喻

**速查卡（答辩评分对策章末尾必须包含）：**
- 三列表格：类别 | 假设提问（短版）| 一句话应答
- 13~20 条，覆盖基础和逐页提问的核心题
- 字号 9pt，紧凑排版，方便打印随身携带

**评分对应方案至少包含：**
- 评分标准原文引用（如有）
- 每项评分维度的对策
- "最容易拿分的 3 个点"和"最容易丢分的 3 个点"

**自检清单至少包含：**
- PPT 页数确认
- 核心功能运行测试
- 讲稿熟悉度
- 备用方案（演示失败时的 Plan B）
