# 课程答辩准备文档生成器

为大学课程设计项目生成一份结构化的答辩准备 DOCX 文档。核心价值：**把项目代码和评分标准转化为答辩时可直接使用的问答手册**。

## 触发方式

| 方式 | 操作 |
|------|------|
| **自动触发** | 对话中提到"答辩文档""答辩攻略""准备答辩""假设提问""defense""defense preparation"等关键词自动加载 |
| **手动命令** | 输入 `/答辩` 或 `/defense` 或 `/course-defense-prep` |

> 手动命令不管当前上下文是什么都会强制执行此 skill。建议在对话开头就输入 `/答辩` 确保 skill 已激活。`,

## 前置检查：协作 Skill 探测

**在开始任何工作前，先检查以下两个 skill 是否可用：**

```javascript
// 检查方法：查看 available_skills 列表中是否包含对应 skill
// docx — 专业 Word 文档生成
// frontend-design — 前端/视觉设计美化
```

| 协作 Skill | 用途 | 若可用 |
|-----------|------|--------|
| `docx` | 专业 DOCX 生成、表格格式化、样式管理 | 阶段 4 的 DOCX 生成全部委托给它，不需要手写 docx-js 脚本 |
| `frontend-design` | 视觉设计、配色方案、排版美化 | 委托它设计封面配色、表格样式、标题层级视觉方案 |

**调用方式：**
- 若 `docx` 可用 → 在阶段 4 使用 `Skill` 工具调用 `docx`，将文档内容和结构要求传给它，由它生成最终 DOCX
- 若 `frontend-design` 可用 → 在阶段 2 设计完成后，用它生成一份"视觉设计指南"（配色、间距、字体层级、表格边框风格），再传给 `docx` skill 执行
- 两者都可用 → **三 skill 联合作战**：本 skill 负责内容策划 + Q&A 设计 → `frontend-design` 负责视觉方案 → `docx` 负责文档生成
- 都不可用 → 回退到手动 docx-js 脚本模式（阶段 4 原始流程）

> **重要**：不要假设 skill 一定存在。每次执行前必须检查 `available_skills` 列表。Skill 工具调用不存在的 skill 会报错。

## 工作流程（4 阶段）

### 阶段 1：收集信息

**必须阅读的文件：**
1. 课程评分标准 / 考查要求文档（通常为 `.docx`）— 提取评分维度、知识点要求、答辩流程
2. 项目全部源代码 — 重点读核心算法、架构设计、数据模型
3. 项目 README.md / CLAUDE.md（如有）— 了解项目背景和技术栈

**提取关键信息：**
- 评分维度及各维度分值
- 课程要求的硬性知识点（如"必须覆盖继承、接口、JDBC"）
- 项目中的关键技术决策和算法
- 项目包结构和分层架构

### 阶段 2：设计文档结构

按以下十大章节组织文档：

```
一、项目概述（30 秒版）     ← 一句话讲清楚做了什么
二、核心知识点详解           ← 每个知识点配本项目代码示例
三、基础概念高频问答         ← 评委爱问的通用概念题
四、核心算法详解             ← 项目特有算法的全链路讲解
五、数据库/数据设计          ← 表结构、设计理由
六、答辩演示流程             ← 分步骤+每步展示点+追问预警
七、项目结构                 ← 包图/分层架构
八、假设提问与回答对应方案   ← 本文档核心，见下方详解
九、答辩评分对应方案         ← 评分标准→对策→分数档对比
十、演示前自检清单           ← 逐项打勾
```

### 阶段 3：编写假设提问与回答对应方案（核心）

这是本文档区别于普通答辩 PPT 的关键。按照 **假设提问 → 标准回答 → 可能追问 → 追问应对** 四段式组织。

**问题分为六大类：**

| 类别 | 考查维度 | 问题数建议 | 示例问题方向 |
|------|---------|-----------|-------------|
| 基础概念类 | 答辩·基本概念 | 5-6 题 | OOP 特性、访问修饰符、static/final、接口vs抽象类 |
| 代码实现类 | 答辩·技术深入 | 5-6 题 | 核心算法、数据结构选择、异常处理、并发控制 |
| 设计思路类 | 答辩·系统设计 | 3-4 题 | 为什么选这个方案、架构分层、解耦方式 |
| 数据层类 | 数据库设计 | 3-4 题 | 表设计、SQL 优化、ORM/原生 JDBC 选择 |
| GUI/前端类 | 系统实现 | 2-3 题 | 组件选择、事件处理、EDT/渲染管线 |
| 开放思考类 | 答辩·综合能力 | 4-5 题 | 如果重做怎么改进、扩展到 N 倍规模、测试策略 |

**追问链设计原则：**
- 追问1 沿着回答中的技术细节深挖一步（"你说的XXX具体怎么实现的？"）
- 追问2 考察边界条件或替代方案（"如果条件变了怎么办？"）
- 追问3 考察语言特性或底层原理（"Java 层面是怎么做到的？"）

**编写要点：**
- 回答必须能关联到项目中的具体代码位置
- 每个回答控制在 3-5 句话，便于临场记忆
- 追问应对要比标准回答更深一层，展示真实理解
- 压力测试题（"太简单了"、"这个有什么技术含量"）必须准备

**附加快查卡：** 在 8.x 末尾放一张表格，每条一行："类别 | 假设提问（短版）| 一句话应答"，方便答辩前 5 分钟快速过一遍。

### 阶段 4：生成 DOCX

**根据前置检查结果选择策略：**

#### 策略 A：docx + frontend-design 双 skill 可用（推荐）

1. **先调 `frontend-design`**：将十大章节的文档大纲传给它，生成一份视觉设计指南，包括：
   - 封面配色方案（标题色、背景色、分割线色）
   - 标题层级字体大小与间距（H1/H2/H3 + 正文 + 代码块）
   - 表格样式（表头底色、边框粗细、单元格 padding）
   - 代码块风格（背景色、字体、行间距）
   - 页眉页脚样式

2. **再调 `docx`**：将视觉设计指南 + 完整文档内容 + 章节结构传给 `docx` skill，由它生成最终的 `.docx` 文件。

#### 策略 B：仅 docx 可用

直接调 `docx` skill，传递完整文档内容和基本格式要求，由它生成 DOCX。

#### 策略 C：回退到手动脚本

若两个 skill 都不可用，使用 `docx` npm 包手写生成脚本。
模板见 `references/docx-template.md`。

**关键注意事项（手动模式下）：**
- JS 字符串中不能使用未转义的 ASCII 双引号 `"`，中文内容中的引号要用 `\"` 转义
- 表宽必须使用 `WidthType.DXA`（不能用 PERCENTAGE，Google Docs 不兼容）
- 代码块使用等宽字体（`Consolas`）+ 灰色背景（`F5F5F5`）模拟 IDE 效果
- 页面设为 A4，字体用 `Microsoft YaHei`（微软雅黑）支持中文
- 生成后检查：所有 Heading 编号正确、表格完整、无乱码

**手动运行命令：**
```bash
cd <project-dir>
NODE_PATH="$(npm root -g)" node scripts/generate-defense-docx.js
```

## 文档质量标准

**回答质量三原则：**
1. **关联代码** — 每个技术回答必须能说"在 XX.java 第 Y 行"
2. **控制长度** — 标准回答 3-5 句，追问应对 2-3 句
3. **递进深度** — 追问应对要比标准回答深一层，不能是换个说法重复

**必须避免的问题：**
- ❌ 回答像背课文 — 要用口语化的解释，带比喻
- ❌ 追问应对 = 标准回答的重述 — 必须是更深的技术细节
- ❌ 代码示例和实际代码不一致 — 直接从源文件拷贝
- ❌ DOCX 中 JS 字符串未转义引号导致语法错误
- ❌ 忘记在第八章末尾放「快速应答速查卡」

**评分对应方案（第九章）至少包含：**
- 评分标准原文引用
- 每项评分维度的对策
- 不同分数段的表现对比表
- "最容易拿分的 3 个点"和"最容易丢分的 3 个点"
